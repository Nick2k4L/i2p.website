## Přehled

Tento dokument specifikuje schéma podpisů s možností opětovné randomizace, které je vhodné pro vytváření Destinations, které mohou být zaslepeny. Navíc může být použito k zaslepení existujících Ed25519 Destinations, s mírným snížením účinnosti.

## Motivace

[Návrh 123](/proposals/123-new-netdb-entries) definuje šifrovaný formát LeaseSet2, který ztělesňuje princip nejmenší autority: každý účastník sítě dostává pouze informace nezbytné pro svou roli. Konkrétně šifrovaný LeaseSet2 publikovaný na floodfill neodhaluje Destination, pro které je určen, a Leases mohou zobrazit pouze ti, kteří mají předchozí znalost Destination. Floodfills však stále potřebují být schopny ověřit šifrované LeaseSet2 při jejich publikování a klienti navíc potřebují zajistit, že ověření bylo vynuceno samotnou Destination.

Návrh 123 toho dosahuje zaslepením podepisovacích klíčů Destinací. Zaslepené klíče lze použít k vytvoření podpisů, které jsou ověřitelné floodfilly, a klienti si mohou být jisti, že pouze Destinace mohla tyto podpisy vytvořit. Je proto nutné specifikovat schéma podpisu, které lze použít pro zaslepení.

## Návrh

### Základní schéma podpisu

Schéma podpisu zde specifikované, Red25519, je implementací [RedDSA](https://github.com/zcash/zips/tree/master/protocol/protocol.pdf), podpisového schématu založeného na Schnorrovi, které podporuje re-randomizaci klíčů. Má následující funkce:

**GENERATE_PRIVATE()** : Vrací rovnoměrně náhodný privátní klíč.

**DERIVE_PUBLIC(sk)** : Vrací veřejný klíč odpovídající danému privátnímu klíči.

**GENERATE_RANDOM()** : Vrací náhodný skalár vhodný pro opětovnou randomizaci páru klíčů.

**RANDOMIZE_PRIVATE(sk, alpha)** : Znovu randomizuje privátní klíč pomocí tajného skaláru alpha.

**RANDOMIZE_PUBLIC(vk, alpha)** : Opětovně randomizuje veřejný klíč pomocí tajného skaláru alpha.

**SIGN(sk, m)** : Vrací podpis vytvořený privátním klíčem sk nad danou zprávou m.

**VERIFY(vk, m, sig)** : Ověřuje podpis sig proti veřejnému klíči vk a zprávě m. Vrací true, pokud je podpis platný, jinak false.

Pro daný pár klíčů (sk, vk) platí následující vztah:

```
RANDOMIZE_PUBLIC(vk, alpha) == DERIVE_PUBLIC(RANDOMIZE_PRIVATE(sk, alpha))
```
### Převod Ed25519 klíčů na Red25519

Ed25519 klíče MOHOU být přechodně jednosměrně převedeny na Red25519 klíče, aby bylo možné podporovat opětovnou randomizaci existujících Ed25519 Destinations. Ostatní sigtypes nejsou kompatibilní.

Definujeme následující konverzní funkce:

**CONVERT_ED25519_PRIVATE(privkey)** : Vrací Red25519 privátní klíč odpovídající danému Ed25519 privátnímu klíči.

**CONVERT_ED25519_PUBLIC(pubkey)** : Vrací Red25519 veřejný klíč odpovídající danému Ed25519 veřejnému klíči.

Pro danou dvojici Ed25519 klíčů (privkey, pubkey) platí následující vztah:

```
CONVERT_ED25519_PUBLIC(pubkey) == DERIVE_PUBLIC(CONVERT_ED25519_PRIVATE(privkey))
```
## Specifikace

### Definice

**B** : Ed25519 základní bod podle [RFC-8032](https://tools.ietf.org/html/rfc8032).

**L** : Řád Ed25519 2^252 + 27742317777372353535851937790883648493 jak je uvedeno v [RFC-8032](https://tools.ietf.org/html/rfc8032).

**[s] B** : Násobení základního bodu s pevnou základnou skalárem s.

**[s] A** : Násobení skaláru A proměnnou základnou s.

**x || y** : Spojit dvě pole bajtů x a y.

### Red25519

Schéma Red25519 specializuje RedDSA s:

- G := skupina bodů na Edwardsově formě křivky Curve25519. Konkrétně to znamená, že Red25519 používá podskupinu prvočíselného řádu L a kofaktor h_G je 8.
- P_G := Ed25519 základní bod B.
- l_H := 512
- H(x) := SHA-512("I2P_Red25519H(x)" || x)

RedDSA předpokládá, že H(x) je implementována s kryptografickou hashovací funkcí, která je bezpečná proti útokům rozšíření délky. SHA-512 toto samo o sobě nesplňuje. Abychom to napravili, vyžadujeme, aby zprávy byly prefixovány prefix-free kódováním jejich délky:

```
len_u16(M) || M
```
kde `len_u16(M)` je 2bajtová reprezentace délky M v little-endian formátu (pro konzistenci s little-endian kódováním skalárů a bodů).

Zprávy nesmí být delší než 65534 bajtů. Délka 65535 je rezervována pro možná budoucí rozšíření.

### Kódování a dekódování

Red25519 privátní klíče jsou skaláry mod L, kódované v little-endian reprezentaci. Definujeme funkce `DECODE_SCALAR` a `ENCODE_SCALAR` pro převod mezi formou bajtového pole a celočíselnou formou skaláru.

Red25519 veřejné klíče jsou body na Edwardsově formě Curve25519. Jsou kódovány jako 255-bitová little-endian reprezentace y-souřadnice, následovaná jediným bitem indikujícím znaménko x-souřadnice. Toto je stejné kódování jako u Ed25519. Definujeme funkce `DECODE_POINT` a `ENCODE_POINT` pro převod mezi formou pole bajtů a souřadnicovou formou bodu.

### Funkce RedDSA

Pro snadnější implementaci níže explicitně vypisujeme funkce RedDSA (stejně jako několik pomocných funkcí) již specializované pro Red25519. Implementátoři by se měli obrátit na sekci 5.4.6 [RedDSA](https://github.com/zcash/zips/tree/master/protocol/protocol.pdf) pro obecnou specifikaci funkcí RedDSA.

```
HStar(prefix1, prefix2, m) :=
    h = SHA-512()
    h.input("I2P_Red25519H(x)")
    h.input(prefix1)
    h.input(prefix2)

    h.input(len(m) & 0xff)
    h.input((len(m) >> 8) & 0xff)
    h.input(m)

    s = h.digest()
    return s mod L

GENERATE_PRIVATE :=
    s = 64 random bytes
    return s mod L

DERIVE_PUBLIC(sk) := [sk] B

GENERATE_RANDOM :=
    s = 64 random bytes
    return s mod L

RANDOMIZE_PRIVATE(sk, alpha) := (sk + alpha) mod L

RANDOMIZE_PUBLIC(vk, alpha) := vk + [alpha] B

SIGN(sk, m) :=
    T = 80 random bytes
    vkBytes = ENCODE_POINT(DERIVE_PUBLIC(sk))
    r = HStar(T, vkBytes, m)

    R = [r] B
    Rbytes = ENCODE_POINT(R)

    c = HStar(Rbytes, vkBytes, m)
    S = (r + (c * sk)) mod L

    return Rbytes || ENCODE_SCALAR(S)

VERIFY(vk, m, sig) :=
    Rbytes = sig[0..32]
    Sbytes = sig[32..64]

    R = DECODE_POINT(Rbytes)
    if R is invalid:
        return false

    S = DECODE_SCALAR(Sbytes)
    if S >= L:
        return false

    vkBytes = ENCODE_POINT(vk)
    c = HStar(Rbytes, vkBytes, m)
    return ((-[S] B) + R + ([c] vk)).multiplyByCofactor().isIdentity()
```
### Konverzní funkce

```
CONVERT_ED25519_PRIVATE(privkey) :=
    s = SHA-512(privkey)[0..32]
    s[0] = s[0] & 248
    s[31] = (s[31] & 63) | 64
    return s

CONVERT_ED25519_PUBLIC(pubkey) := pubkey
```
Poznámka: implementace `CONVERT_ED25519_PRIVATE` je ekvivalentní k výpočtu tajného skaláru s při odvození Ed25519 veřejného klíče z Ed25519 soukromého klíče, jak je specifikováno v krocích 1-3 z části 5.1.5 [RFC-8032](https://tools.ietf.org/html/rfc8032).

## Bezpečnostní důsledky

Opětovná randomizace Red25519 Destination a následné vytváření podpisů s ní neprozrazuje žádné informace o Destination, protože distribuce Red25519 privátních klíčů generovaných pomocí `RANDOMIZE_PRIVATE` je identická s distribucí privátních klíčů generovaných pomocí `GENERATE_PRIVATE` a `DERIVE_PUBLIC` je deterministické.

Převod soukromých klíčů Ed25519 na Red25519 prostřednictvím `CONVERT_ED25519_PRIVATE` nevede ke stejnému rozložení. Nicméně považujeme snížení bezpečnosti za přijatelné z následujících důvodů:

- Prostor Ed25519 skalárů je zhruba poloviční oproti prostoru Red25519 skalárů (existuje 2^251 možných Ed25519 skalárů a L ~= 2^252 možných Red25519 skalárů). Ztráta bezpečnosti je tedy nanejvýš faktor zhruba 2, nebo zhruba 1 bit (protože jsme mohli náhodou zvolit Red25519 skalár, který je také platným Ed25519 skalárem).
- Existující Ed25519 Destinations již byly historicky vystaveny v síti a je třeba předpokládat, že škodlivé floodfills je již vyjmenovaly.

Uživatelé, kteří se obávají tohoto snížení bezpečnosti, by měli místo Ed25519 používat Red25519 jako sigtype pro své Destinations.

Všimněte si, že výše uvedený argument se nevztahuje na skalár pro opětovnou randomizaci alpha; informace o klíči unikají pokaždé, když je vybrána zaujatá alpha, protože aditivní opětovná randomizace se chová jako jednorázová šifra.

## Kompatibilita

Verze I2P, které podporují Red25519, budou schopny ověřovat síťové datové struktury podepsané pomocí něj. Verze I2P, které Red25519 nepodporují, s ním budou zacházet jako s neznámým podpisem a MOHOU tyto datové struktury zahodit. Uživatelé by měli očekávat, že spolehlivost datových struktur podepsaných Red25519 bude nízká, dokud nebude dosaženo dostatečného nasazení Red25519.
